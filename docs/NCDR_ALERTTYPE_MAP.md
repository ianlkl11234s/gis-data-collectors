> 結論先講：**既有 `collectors/ncdr_alerts.py`（總和 feed、15 分鐘輪詢）已涵蓋所有類型，
> 包含停電（電力中斷）、停水、火災、行動電話中斷等，不需要新增 collector。**
> 本文件記錄 AlertType 1–61 掃描結果與 feed 留存行為，供日後查考。

## 1. 兩種 feed 的差異

| | Per-type feed | 總和 feed（collector 現用） |
|---|---|---|
| URL | `RssAtomFeed.ashx?AlertType={n}`（另有 `JSONAtomFeed.ashx`） | `JSONAtomFeeds.ashx`（注意複數 Feeds） |
| 內容 | 該類型「近期」事件 | **所有類型**「目前生效中」事件 |
| 類型數 | 只有 **12 個有效編號**（見下表） | 實測 2 個月累積 **27 種**（理論上全部 61+ 種） |
| 留存 | 高量類型 ~7 天滾動窗；低量類型保留最後數筆（颱風 feed 還留著 211 天前的） | 過期即消失（依 CAP `expires`） |
| 認證 | 匿名、Public Domain | 匿名、Public Domain |
| 限流 | 有（連續 request 會 429，間隔 ≥3.5s 即穩定） | 同左 |

**總和 feed 是更優來源**：per-type feed 缺停水/火災/停電/海洋污染/空品/停班停課以外的多數民生類型。

## 4. 每則訊息存在多久（決定輪詢頻率）

各類型生效時長（`expires - effective`）統計：

- 中位數最短的也有 60 分鐘（地震、火災、空品）；停水中位數 6.5 小時；多數 2 小時以上
- **生效 < 30 分鐘的只有 17/4,416 筆（0.4%）**，幾乎全是短時降雨示警
- → **15 分鐘輪詢漏接率 ≈ 0**，30 分鐘也大致安全；現行 15 分鐘維持即可

風險備註：總和 feed 是「目前生效中」快照，若 collector 停機超過某示警的生效窗就永久漏掉。
若要補救可另用 per-type feed（7 天回溯）回填 12 個主要類型，目前未實作（停機超過 1 小時
的情況罕見，且 Zeabur 重啟後立即補跑）。
